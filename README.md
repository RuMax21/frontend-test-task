# Frontend Test Task

Тестовое задание
Стек: React + Typescript + scss + RTK, бандлер - Vite

- **`/`** — Задание 1: мини-каталог товаров с поиском, фильтром по категории и корзиной на Redux Toolkit.
- **`/team`** — Задание 2: список участников команды, форма добавления, модальное окно с деталями, удаление с отменой.

## Запуск через Docker

```bash
docker build -t mini-store-team .
docker run -p 3000:3000 mini-store-team
```

Приложение будет доступно на `http://localhost:3000`.

## Что реализовано

### Задание 1 — Mini Store (`/`)

- Каталог товаров с карточками (название, цена, категория, кнопка «В корзину»).
- Поиск по названию и фильтр по категории, синхронизированы с query-параметрами URL (`?search=...&category=...`) — фильтры восстанавливаются после перезагрузки страницы.
- Корзина: изменение количества, удаление позиции, итоговая сумма.
- Удаление товара из корзины происходит мгновенно, с уведомлением и кнопкой «Отменить», которая возвращает позицию на прежнее место.
- Корзина сохраняется в `localStorage` и восстанавливается при перезагрузке.
- Состояние каталога, фильтров и корзины хранится в Redux Toolkit store: `productsSlice` (поиск/категория), `cartSlice` (корзина), плюс `productApi`/`categoryApi` как RTK Query API-слайсы для моковых данных.
- Persist корзины в `localStorage` реализован через `listener middleware`, а не через ручной `useEffect` в компонентах — так побочный эффект живёт рядом со слайсом.

### Задание 2 — Project Team (`/team`)

- Список участников в виде карточек: имя, должность, email, статус `active`/`inactive`, кнопка «Подробнее».
- Модальное окно с полной информацией об участнике, закрывается по клику вне окна, по Escape и по кнопке.
- Форма добавления участника на `react-hook-form` + `zod`: контролируемые поля, валидация (имя и должность обязательны, email проверяется по формату), понятные сообщения об ошибках.
- Удаление участника — карточка исчезает сразу, появляется уведомление с кнопкой «Отменить», отмена возвращает участника на прежнюю позицию в списке.
- Список участников сохраняется в `localStorage` и восстанавливается после перезагрузки.
- Состояния: пустой список, ошибки валидации, hover/active для кнопок и полей.
- Список участников и все действия над ним (`add`/`remove`/`undo`) реализованы на локальном состоянии React, как и указано в задании. RTK Query здесь используется только один раз — как имитация сервера, от которого подтягиваются моковые участники при самом первом визите, дальше вся логика живёт вне store.

## Какие решения были приняты

- **RTK Query с `fakeBaseQuery`** вместо простого массива в файле — данные каталога и моковых пользователей получаются через `useGetProductsQuery`/`useGetUsersQuery` с искусственной задержкой, это ближе к реальному сценарию и не потребовало бы переписывания компонентов при подключении настоящего API.
- **`listener middleware`** для персиста в `localStorage` — синхронизация со storage вынесена из компонентов в отдельный слушатель рядом со слайсом.
- **Sonner** для уведомлений об удалении — единый паттерн и в корзине, и в команде.

## Что бы улучшили при наличии времени

- **Обобщить optimistic delete + undo.** Сейчас он реализован дважды и по-разному: в `cartSlice` — через Redux, в команде — через `useState` в `useTeamUsers`. По сути это один и тот же UX-паттерн, и его стоило бы вынести в переиспользуемый примитив. Это снизило бы дублирование и защитило бы от рассинхронизации поведения между двумя фичами.
- **Реальная accessibility-проходка.** Модалка сейчас закрывается по Escape/оверлею, но не делает focus trap и не возвращает фокус на элемент, который её открыл; уведомления от Sonner не размечены как `aria-live`.
- **Производительность на масштабе.** Каталог и список участников сейчас рендерят весь список плоско — нормально для 6-20 карточек из задания, но при реальных сотнях товаров потребовалась бы виртуализация списка и мемоизация карточек, чтобы фильтрация по поиску не перерисовывала весь грид на каждый символ.
- **Единый источник правды для валидации.** Zod-схема в `user-form` сейчас существует только для формы. В реальном продукте её стоило бы переиспользовать и на уровне типов API-ответа, чтобы форма и бэкенд не могли разойтись по составу полей независимо друг от друга.
